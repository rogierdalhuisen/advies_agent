# Orchestrator Graph Dataflow Architecture

This document explains the complete dataflow and state management within the `orchestrator_graph` module (`@src/graph/orchestrator_graph/**`). The system uses LangGraph to orchestrate a hierarchical, multi-agent workflow for generating insurance recommendations.

---

## 1. Global State (`OrchestratorState`)
The entire process is driven by a shared state dictionary (`OrchestratorState`) that gets mutated or appended to by each node. 

**Initial Inputs (Provided at invocation):**
*   `user_profile`: Raw client profile data from an advice form.
*   `premiums`: Pre-calculated premiums (`provider -> coverage_level -> amounts`).
*   `available_providers`: List of provider names to consider.
*   `product_descriptions`: Static markdown descriptions for each provider.

**State Fields Updated During Execution:**
*   `parsed_constraints`: Structured interpretation of the user's profile and needs.
*   `retrieval_tracker`: A complex dictionary tracking the status (`active`, `pending`, `retrieved`, `dropped`) of every provider, coverage level, and specific aspect (topic) being investigated.
*   `retrieval_tasks`: A list of search tasks (provider + query + aspect) dynamically generated by the orchestrator.
*   `retrieval_summaries`: A list of results from parallel retriever runs, accumulated using an `operator.add` reducer.
*   `retrieval_iteration`: Counter to limit retrieval loops (max 2 iterations as per `route_after_assessment`).
*   `orchestrator_notes`: A log of the orchestrator's decision reasoning across iterations.
*   `qualitative_assessment`: Intermediate evaluation of active insurance options.
*   `recommendation`: The final output containing top and budget recommendations.

---

## 2. Node-by-Node Dataflow

The graph execution follows this path: `START -> user_agent -> build_tracker -> orchestrator_assess -> [conditional loop or end]`.

### A. Initialization & Setup

#### Node 1: `user_agent` (`user_agent_node`)
*   **Reads from state:** `user_profile`
*   **Action:** Uses an LLM (Claude Opus) to parse the raw user profile into a structured `ParsedConstraints` object. It extracts core data, interprets explicit/implicit needs, notes context/trade-offs, and generates specific English `retrieval_aspects` (topics to investigate).
*   **Writes to state:** `{"parsed_constraints": ...}`

#### Node 2: `build_tracker` (`build_initial_tracker_node`)
*   **Reads from state:** `parsed_constraints`, `premiums`, `available_providers`
*   **Action:** Pure Python function. It builds the skeleton of the `retrieval_tracker`. For every available provider, it marks their coverage levels (based on `premiums`) as `active`, and creates an entry for every `retrieval_aspect` generated by the `user_agent`, initially marking them all as `pending`.
*   **Writes to state:** Initializes `retrieval_tracker`, sets `retrieval_iteration` to `0`, clears `retrieval_tasks` and `orchestrator_notes`.

---

### B. The Orchestration Loop

The system enters a loop between assessing what is known, fetching what is missing, and merging the results.

#### Node 3: `orchestrator_assess` (`orchestrator_assess_node`)
*   **Reads from state:** `parsed_constraints`, `product_descriptions`, `premiums`, `retrieval_tracker`, `retrieval_iteration`, `orchestrator_notes`
*   **Action:** Acts as the brain. Uses an LLM (Claude Opus) to review the `retrieval_tracker`. 
    *   It drops providers or specific coverage levels that are clearly irrelevant.
    *   It generates new `retrieval_tasks` for aspects that are still `pending` for `active` providers.
*   **Writes to state:** Updates the `status` of providers/coverage levels in the `retrieval_tracker`, outputs a new list of `retrieval_tasks`, and appends reasoning to `orchestrator_notes`.
*   **Conditional Routing (`_dispatch_or_evaluate`):** 
    *   If `retrieval_tasks` exist AND `retrieval_iteration` < 2: Proceeds to **Fan-out Retriever** via LangGraph's `Send()` API.
    *   Otherwise: Breaks the loop and proceeds to **Evaluation (`evaluator_step1`)**.

#### Node 4: `retriever` (Subgraph invoked via `Send` parallel fan-out)
For every task in `retrieval_tasks`, a separate instance of the `retriever_agent` subgraph is spun up in parallel.
*   **Input (Mapped from `Send`):** `provider`, `query`, `aspect`, `product_description`
*   **Subgraph Internal Flow (`RetrieverAgent`):**
    1.  **`retrieve`:** Fetches documents using `InsuranceRetriever` (applies dynamic `k` boosting for pregnancy keywords).
    2.  **`rerank`:** Reranks documents using `Reranker`.
    3.  **`verify`:** Uses a lightweight LLM (Gemini Flash) to check if the docs fully answer the query. Returns `complete`, `partial` (with `missing_info`), or `miss`.
    4.  **Conditional Routing:** 
        *   If `complete` or `miss` or max retries hit -> Go to `summarize`.
        *   If `partial` -> Go to `save_documents` -> `rewrite` (rewrites query using `missing_info` via GPT-4o-mini) -> loop back to `retrieve`.
    5.  **`summarize`:** Uses an LLM (GPT-4o) to generate a structured `RetrievalSummary` outlining findings per coverage level.
*   **Writes to Parent State:** Each parallel run returns `{"retrieval_summaries": [summary]}`. The parent state uses an `operator.add` reducer to concatenate all these lists into a single array.

#### Node 5: `merge_results` (`_merge_retriever_results`)
*   **Reads from state:** `retrieval_tracker`, accumulated `retrieval_summaries`, `retrieval_iteration`
*   **Action:** Pure Python function. Iterates over all summaries returned by the parallel retrievers and injects them into the corresponding provider -> aspect path in the `retrieval_tracker`. It marks that aspect's status as `retrieved`.
*   **Writes to state:** Updated `retrieval_tracker`, increments `retrieval_iteration` (+1), and clears `retrieval_summaries` to `[]` for the next loop.
*   **Routing:** Unconditionally loops back to `orchestrator_assess`.

---

### C. Evaluation & Final Output

Once the orchestrator decides enough information has been gathered (or iteration limits are reached), it moves to the tradeoff agents.

#### Node 6: `evaluator_step1` (`evaluator_step1_node`)
*   **Reads from state:** `parsed_constraints`, `retrieval_tracker` (filtered to ONLY include `active` providers and coverage levels), `premiums`, `product_descriptions`
*   **Action:** Uses an LLM (GPT-4o) to produce a `QualitativeAssessment`. It looks at the aggregated summaries and evaluates the overall fit, strengths, weaknesses, and uncertainties for each active provider-coverage combination individually. It also notes cross-provider patterns.
*   **Writes to state:** `{"qualitative_assessment": ...}`

#### Node 7: `evaluator_step2` (`evaluator_step2_node`)
*   **Reads from state:** `parsed_constraints`, `retrieval_tracker` (filtered), `premiums`, `product_descriptions`, `qualitative_assessment`
*   **Action:** Uses an LLM (GPT-4o) acting as a senior advisor. It takes the qualitative breakdown and makes the final hard choices. It outputs a `FinalRecommendation` containing 2 top-fit options, 2 budget options, notable rejected options, and an overall analysis.
*   **Writes to state:** `{"recommendation": ...}`
*   **Routing:** Goes to `END`.
